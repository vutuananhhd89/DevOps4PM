## 💡 The Core Insight

Yesterday's digest (Feb 10) landed on a critical truth: **the bottleneck has shifted to PMs** — to deciding *what* to build, fast and clearly.

Today's follow-up answers the obvious next question: **How do you write specs that AI agents can actually execute?**

Two opposing philosophies have emerged:

- **Vibe Coding** — prompt an AI with loose instructions, iterate live, "feel" your way to the solution
- **Spec-Driven Development (SDD)** — write precise, structured specifications *before* AI touches a line of code

The industry is learning — painfully — that **vibe coding works for prototypes but breaks in production.** And the PM who can write an executable spec is now more valuable than the PM who writes a beautiful PRD.

---

## 🎯 Vibe Coding vs. Spec-Driven Development

### Side-by-Side Comparison

| Dimension | Vibe Coding | Spec-Driven Development |
|-----------|-------------|-------------------------|
| **Philosophy** | "Just tell the AI what you want" | "Define exactly what you need first" |
| **Speed** | ⚡ Fast start, slow finish | 🐢 Slow start, fast finish |
| **Input** | Loose prompts, natural language | Structured specs, acceptance criteria |
| **Best for** | Prototypes, MVPs, exploration | Production systems, team projects |
| **Risk** | Messy code, hidden bugs, tech debt | Over-engineering, analysis paralysis |
| **PM role** | "Try this, try that" | "Here's the contract — build it" |
| **Debugging** | Hard — decisions buried in chat history | Easy — spec is the source of truth |
| **Scalability** | ❌ Breaks past 1 person | ✅ Scales across teams and agents |

### When to Use Each

```
VIBE CODING ⚡                          SPEC-DRIVEN 📋
├── Hackathons                          ├── Production features
├── Personal projects                   ├── Team collaboration
├── Proof-of-concept                    ├── Regulated industries
├── Learning new frameworks             ├── Multi-agent workflows
├── Quick demos                         ├── Long-term maintainability
└── "Can this even work?"               └── "This MUST work reliably"
```

> **The key insight:** It's not either/or — it's a spectrum. Start with vibe coding to explore, switch to spec-driven when you're ready to ship.

## 📋 What Is an "Executable Spec"?

Traditional specs (PRDs) are written for **humans** — long, narrative, open to interpretation.

Executable specs are written for **AI agents** — precise, structured, testable.

### Traditional PRD vs. Executable Spec

| Aspect | Traditional PRD | Executable Spec |
|--------|-----------------|-----------------|
| **Audience** | Engineers + designers | AI agents + engineers |
| **Format** | Prose paragraphs | Structured YAML/Markdown sections |
| **Ambiguity** | "Should handle errors gracefully" | "Return 422 with `{ error: 'invalid_input', field: 'email' }` when email format fails regex `^[a-zA-Z0-9+_.-]+@[a-zA-Z0-9.-]+$`" |
| **Success criteria** | "Works well" | Measurable acceptance tests |
| **Length** | 5-20 pages | 1-3 pages, dense |
| **Maintenance** | Stale within a sprint | Living document, versioned |

### Anatomy of an Executable Spec

```markdown
# Feature: User Onboarding Flow

## Purpose
Reduce time-to-first-value from 12 minutes to under 3 minutes.

## Interfaces
- Input: user email, name, role (dropdown: PM, Engineer, Designer)
- Output: personalized dashboard with 3 recommended actions

## Constraints
- Must work offline-first (PWA)
- Max 2 API calls during onboarding
- Accessibility: WCAG 2.1 AA compliant

## Edge Cases
- Duplicate email → show "Already registered" with login link
- Network timeout → save progress locally, retry on reconnect
- Empty role → default to "PM" with option to change

## Acceptance Criteria
- [ ] User completes onboarding in < 3 min (p90)
- [ ] Dashboard loads in < 1.5s on 3G connection
- [ ] Zero data loss on network interruption

## Evaluation Rubric (for AI agent)
- Correctness: Does it implement all edge cases? (weight: 40%)
- Performance: Meets latency targets? (weight: 30%)
- Code quality: Clean, testable, documented? (weight: 30%)
```

## 🔄 The PM's New Workflow: Spec → Ship → Learn

### Old PM Workflow (Pre-AI)

```
Research (2 weeks) → PRD (1 week) → Design (2 weeks)
→ Eng Build (4-6 weeks) → QA (1-2 weeks) → Launch
Total: ~12 weeks
```

### New PM Workflow (Spec-Driven + AI)

```
Problem Definition (2-3 days) → Executable Spec (1 day)
→ AI Generates Prototype (hours) → Human Review (1 day)
→ Iterate Spec (1 day) → Ship (same week)
Total: ~1-2 weeks
```

### What Changes for PMs

| Old Habit | New Habit |
|-----------|-----------|
| Write long PRDs | Write tight, parseable specs |
| Describe *what it looks like* | Describe *what it does* — inputs, outputs, edge cases |
| Verbal handoff to engineers | Spec IS the handoff |
| Review mockups | Review working prototypes |
| Sprint planning meetings | Continuous spec refinement |
| "Let's discuss in the next sync" | "Update the spec, re-run the agent" |

## ⚠️ The Traps to Avoid

| Trap | What Happens | Fix |
|------|-------------|-----|
| **All vibe, no spec** | Ship fast, break everything, accumulate AI-generated tech debt | Time-box exploration, then spec it |
| **Over-specifying** | Spend 2 weeks writing specs for a feature nobody wants | Validate the problem first, spec only what's validated |
| **Spec rigidity** | Treat specs as sacred — refuse to iterate | Specs are living documents, not contracts |
| **Ignoring AI limits** | Assume AI understands context perfectly | Include explicit context, constraints, and "what NOT to do" |
| **Skipping evaluation** | No acceptance criteria = no way to know if AI built it right | Every spec needs testable success criteria |

## 🧠 The Maturity Model: From Vibe to Spec-Native

```
Level 1: Vibe Coder
  "Build me a dashboard" → AI makes something → manual iteration
         ↓
Level 2: Prompted Builder
  Detailed prompts with constraints → AI builds closer to target
         ↓
Level 3: Spec Writer
  Structured specs with edge cases → AI builds accurately
         ↓
Level 4: Spec-Native PM
  Specs as living artifacts → AI agents auto-generate, test, deploy
  → Continuous feedback loop, human reviews output not process
```

> Most PMs are at **Level 1-2**. The ones moving to Level 3-4 are becoming the new 10x PMs — not because they code, but because they **specify with precision**.
