### 🎯 Core Concept

**You learned to write specs for AI agents (Feb 11). But here's what ThoughtWorks found when they actually tried: the tools overpromise and the specs under-deliver — unless you understand the craft underneath.**

Böckeler tested three SDD tools and concluded with a German word: **"Verschlimmbesserung"** — making something worse by trying to make it better.

The real skill isn't which tool you pick. It's **how you structure context for the AI** — what Böckeler now calls **"Context Engineering."**

---

### 🔑 Key Takeaways

#### 1. SDD Has Three Levels (Most People Only Know the First)

| Level | What It Means | Tool Example |
|-------|--------------|--------------|
| **Spec-first** | Write a detailed spec before coding with AI | Kiro |
| **Spec-anchored** | Keep the spec alive for evolution & maintenance after the task | Spec-kit (aspires) |
| **Spec-as-source** | Spec IS the source file — humans never touch code | Tessl (`// GENERATED FROM SPEC - DO NOT EDIT`) |

> Most PMs think SDD = "write a spec first." That's only Level 1. Levels 2-3 are where it gets interesting — and risky.

#### 2. The Three Tools Compared

| | **Kiro** (AWS) | **Spec-kit** (GitHub) | **Tessl** |
|---|---|---|---|
| **Workflow** | Requirements → Design → Tasks | Constitution → Specify → Plan → Tasks | Spec → Build → Test |
| **Complexity** | Lightest — 3 markdown files | Heaviest — many files, checklists, branching | Medium — 1:1 spec-to-code mapping |
| **Memory bank** | "Steering" (product.md, structure.md, tech.md) | "Constitution" — immutable high-level rules | Spec IS the artifact |
| **Best for** | Small features, greenfield | Larger features, team governance | Code components, brownfield |
| **Level** | Spec-first only | Spec-first (claims spec-anchored) | Spec-anchored → spec-as-source |

#### 3. What ThoughtWorks Found When They Actually Used These

Böckeler's honest assessment after trying all three:

| Finding | What Happened |
|---------|--------------|
| **"Sledgehammer for a nut"** | Kiro turned a small bug fix into 4 user stories with 16 acceptance criteria |
| **Review overload** | Spec-kit created so many markdown files that reviewing them took longer than just coding the feature |
| **False sense of control** | Even with elaborate specs, the AI agent ignored instructions — duplicated existing classes, over-applied rules |
| **Functional vs. technical confusion** | "Purely functional" spec boundaries were unclear — PMs and devs kept mixing requirement levels |
| **"I'd rather review code"** | Böckeler's direct quote — verbose markdown specs were more tedious than reviewing actual code |

> **The meta-lesson**: More spec ≠ better spec. The AI's context window is bigger, but that doesn't mean it processes everything in it.

#### 4. Context Engineering > Prompt Engineering

Böckeler's February 2026 follow-up article reframes the skill entirely:

| Concept | What It Means |
|---------|--------------|
| **Instructions** | Prompts that tell the agent to DO something: "Write an E2E test in the following way…" |
| **Guidance** | General conventions to FOLLOW: "Always write tests that are independent of each other" |
| **Context interfaces** | How the AI gets MORE context: tools, MCP servers, skills |
| **Who loads context?** | LLM decides (risky — non-deterministic), Human invokes (controlled but manual), Agent software triggers (deterministic) |

**The key insight**: Keep context as small as possible. Build rules files gradually. Even though context windows are technically huge, **agent effectiveness goes DOWN with too much context.**

> "My recommendation would be to build context like rules files up gradually, and not pump too much stuff in there right from the start."
> — Birgitta Böckeler

#### 5. The MDD Warning: We've Tried This Before

Böckeler draws a critical parallel to **Model-Driven Development (MDD)** from 15+ years ago:

| Then (MDD) | Now (SDD) |
|-----------|-----------|
| Models (UML/DSL) → Code generators | Specs (Markdown) → LLM code generation |
| Strict, parseable spec language | Flexible natural language specs |
| Deterministic output | Non-deterministic output |
| Failed for business apps — awkward abstraction level | TBD — LLMs reduce overhead but add uncertainty |

> "I wonder if spec-as-source might end up with the downsides of both MDD and LLMs: Inflexibility and non-determinism."

### ⚠️ Risks & Pitfalls

#### The "Verschlimmbesserung" Trap

- Elaborate SDD workflows create **review overload** — more markdown to review, not less work
- **Fix**: Match spec effort to problem size. Small bug? Skip the framework. Production feature? Spec it properly

#### False Sense of Control

- Checklists, templates, and structured workflows don't guarantee the AI will follow them
- **Fix**: Small, iterative steps beat big up-front specs — always

#### The "Spec = Prompt" Confusion

- "Spec" is already semantically diffused — people use it to mean "detailed prompt"
- **Fix**: Specs are structured, behavior-oriented artifacts. Prompts are one-off instructions. Don't confuse them

### 🛠️ Practical Implementation

**Getting Started (adapted from Böckeler's recommendations):**

1. **Start spec-first, not spec-as-source** — write a focused spec BEFORE prompting the AI, but don't try to make the spec the permanent artifact yet
2. **Build your "memory bank" gradually** — start with one rules file (product context + conventions), expand only when you see gaps
3. **Keep context small** — resist the urge to dump everything into the AI's window. Smaller context = better results
4. **Separate instructions from guidance** — "Do this task" (instruction) vs. "Always follow this rule" (guidance). Keep them in different files
5. **Review the CODE, not just the spec** — Böckeler's warning: reviewing markdown doesn't replace reviewing what the AI actually built

### 💡 Why This Matters for Project Managers (Offshore/Distributed Teams)

1. **Spec-first IS valuable for distributed teams** — a well-structured spec reduces timezone misunderstandings. But DON'T over-spec — the review burden cancels the benefit
2. **"Constitution" / rules files standardize cross-team output** — shared conventions mean AI-generated code from any team member follows the same patterns
3. **Context engineering beats prompt engineering** — teaching your team to structure project context (memory banks, rules files) has more leverage than teaching them "better prompts"
4. **Small, iterative specs beat big up-front specs** — Böckeler's finding confirms what agile taught us. Don't let SDD tools drag you back to waterfall-style requirements
5. **The "who loads context" question is critical for async teams** — if the LLM decides when to load context, it might not load it. For distributed handoffs, use deterministic context triggers (hooks, not hopes)
