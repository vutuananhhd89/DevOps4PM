Spec-Driven Development -- Just write a spec. The AI will do the rest.

A Thoughtworks Engineer, Birgitta Böckeler actually tried 3 SDD tools: Kiro (AWS), Speckit (Github) & Tessl.

The result?

**Verschlimmbesserung.**
(A German word meaning — making something worse by trying to make it better.)

🔨 Kiro (AWS) turned a small bug fix into 4 user stories with 16 acceptance criteria.
📄 Spec-kit (GitHub) created so many markdown files, not less work.
🎲 Tessl generated different code from the same spec every time.

"I'd rather review code than all these markdown files"

**SDD is not one thing.** It has 3 levels:

1️⃣ Spec-first
→ Write a spec before AI codes. (Good habit ✅)

2️⃣ Spec-anchored
→ Keep the spec alive for maintenance. (Experimental ⚠️)

3️⃣ Spec-as-source
→ Humans never touch code. (Risky bet 🚨)

The question is: which level?

And here's the warning nobody talks about:
We tried similar thing before -- Model-Driven Development (MDD).

UML → code generators.
Same promise.
Failed for business apps.

SDD with LLMs removes some overhead.
But adds non-determinism.

**The move:**

👉 Use spec-first as a habit
👉 Skip the SDD tools for now -- a good AGENTS.md is enough
👉 Match spec effort to problem size
👉 Review the CODE, not just the artifacts

Still writing specs nobody reads? Or reviewing the code that actually ships?
